# 安全策略

## 威胁模型与部署安全

ArcartX 在**服务端插件**与**客户端 Mod**之间使用自定义协议通信，该协议在 Minecraft 插件消息通道（`arcartx:main`）之上叠加了 DH 密钥协商 + AES-GCM 加密。

请在部署与评估时理解以下边界：

- **本加密层提供机密性、防篡改与协议混淆，但依赖底层 Minecraft 连接完成端点认证。**
  在 **online-mode（正版验证）** 下，Minecraft 协议加密与 Mojang 会话认证已覆盖整条客户端↔服务端链路（包含全部插件消息），ArcartX 的加密在此基础上属于**纵深防御**。
- **DH 公钥交换本身未做身份认证**（双方交换裸公钥，无签名/证书锁定）。因此在以下场景理论上存在中间人替换密钥的风险：
  - **offline-mode（离线服）**：Minecraft 协议加密未启用；
  - **BungeeCord / Velocity 等代理的后端腿**：代理↔后端流量通常为明文，依赖内网信任。
- 协议中包含向客户端下发脚本/指令的能力（如脚本执行包）。在上述未加密链路上，应将 ArcartX 的协议内容视同明文对待。

### 部署建议

- 生产环境请使用 **online-mode = true**。
- 使用代理（BungeeCord / Velocity）时，确保**代理与后端服务器之间**处于受信任网络（内网隔离 / 防火墙），或为该链路启用传输层加密。
- 不要将 ArcartX 协议层视为独立的、可对抗主动网络攻击者的认证安全边界——它的安全性建立在底层 Minecraft 连接之上。

## 加密说明

- 对称加密：**AES/GCM/NoPadding**（随机 12 字节 IV，128 位认证标签）。
- 密钥协商：**有限域 DH，2048 位**；会话 AES 密钥由 DH 共享密钥派生。
- 自 **2.0.0** 起为破坏性升级（旧版 AES/ECB 客户端不兼容），服务端与客户端 Mod 必须同时升级。
